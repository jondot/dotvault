Team-friendly

Zero secrets on disk

Secrets are resolved at runtime from your vault of choice. Nothing to leak, nothing to rotate in files.

How it works

Config → Resolve → Inject

One config file declares what secrets you need. dotvault resolves them at runtime and injects them into your process.

Step 1

Declare

.dotvault.toml

Map env var names to provider references. Commit this to your repo.

→
Step 2

Resolve

dv run

dotvault fetches each secret from its provider at runtime. Nothing hits disk.

→
Step 3

Inject

env vars

Secrets appear as environment variables in your subprocess. Your code doesn't change.

development
[secrets]
API_KEY = { provider = "keychain", ref = "my-key" }
DB_URL = { provider = "env", ref = "DEV_DB" }
production
[production.secrets]
API_KEY = { provider = "env", ref = "API_KEY" }
DB_URL = { provider = "env", ref = "DATABASE_URL" }

Same config, same dv run command. Environment is detected automatically.

Quick Start

Up and running in 60 seconds

Four commands from zero to secrets-in-your-process.

1

Initialize your project

Creates a .dotvault.toml config file in your project root.

$ dv init
2

Store a secret

Write a secret value to your vault of choice. Here we use the OS keychain.

$ dv put --provider keychain --ref my-api-key --value "sk-..."
3

Map it to an env var

Tell dotvault which env var name maps to which secret reference.

$ dv add --name API_KEY --provider keychain --ref my-api-key
4

Run with secrets

Secrets are resolved and injected. Your code reads normal env vars.

$ dv run -- npm start
